User Friendly System Call Interface For Enhanced Security

A secure abstraction layer that lets applications perform critical system operations through a controlled, human-friendly interface. Reduce complexity for developers while enforcing strict security policies at the OS boundary.

GoalSimplify dangerous system calls into safe, high-level operations.
CorePolicy-driven sandboxed execution for every request.
FocusLeast privilege, auditability, and secure defaults.
secure_syscall_demo.c
Live · Safe mode
int main() {
  struct syscall_request req = {
    .operation = "read_file",
    .path      = "/var/app/config.yaml",
    .flags     = SAFE_ONLY
  };

  int result = uif_secure_call(&req);

  if (result != 0) {
    log_error("Policy denied: config.yaml");
  }

  return 0;
}
Project Overview
What this interface solves
Problem statement

Traditional system calls are powerful but low-level and error-prone. One incorrect flag, unchecked pointer, or missing validation can lead to privilege escalation, data leakage, or system compromise. Most applications re-implement similar checks repeatedly, resulting in inconsistent and fragile security.

This project introduces a user friendly system call interface that sits between applications and the operating system. It offers simple, high-level operations instead of raw syscalls, and enforces centrally managed security policies before any request is allowed to reach the kernel.

System Architecture
End-to-end flow from app to kernel
Design

The interface is built as a thin, language-agnostic middleware that exposes safe APIs to applications and translates them into validated system calls. Every request passes through four core layers.

Layer 1 · Client
User API Adapter

Simple function calls, CLI, or REST endpoints that wrap complex system operations into human readable actions.

Layer 2 · Guard
Validation & Sanitization

Validates parameters, normalizes file paths, enforces size limits, and rejects malformed or risky requests.

Layer 3 · Policy
Security Policy Engine

Applies rules based on user role, time, resource, and action type before any system call is executed.

Security Model
How security is enhanced
Security

The design assumes that any application may be partially compromised. The interface therefore enforces strict controls even for trusted clients, turning dangerous system calls into safe, observable operations.

  • Role-based access control for each system operation (file, process, network, IPC).
  • Static allow-lists for files, directories, devices, and network destinations.
  • Rate limiting and throttling for sensitive actions like process spawning or privileged I/O.
  • Secure error messages that avoid leaking internal file paths or kernel details.
  • Structured audit logs with user, resource, action, decision, and timestamp.
